Overview

NewMarket is a hyperlocal digital community hub covering three content verticals: job postings, real estate listings, and local news. It fills the gap between national platforms (too broad to feel local) and informal WhatsApp groups (no search, no structure, no history). The platform supports bilingual content in English and Hindi, anonymous community posting, nested threaded comments, and a role-based permission system enforced at the database layer.

The platform is built for daily active use, not occasional browsing. This shaped every technical decision toward speed, reliability, and mobile performance. All listing queries return in under 200ms at 100K+ rows. The platform has sustained 99.2% uptime since launch on a zero-ops managed infrastructure.

The Problem

Local communities in Tier 2 and Tier 3 Indian cities lack a structured digital presence. Job opportunities are shared informally in WhatsApp groups that are impossible to search or browse. Property listings are scattered across physical notice boards, regional papers, and unstructured chat threads. Local news either doesn't exist online or lives on slow, ad-saturated regional portals.

Any single vertical could be served by an existing national platform. But the hyperlocal value proposition (everything for this community in one place) required building all three together, making the platform a daily-use destination rather than an occasional utility.

Scale constraint: The target audience uses mid-range Android devices on 4G connections. Every additional 100ms of query time is perceptible. The sub-200ms target wasn't a stretch goal. It was the performance floor for the platform to feel usable.

Key Design Decisions

Composite index strategy: (type, location, created_at) covers the dominant browse-by-type-and-location query, enabling sub-200ms response at 100K+ rows
Partial indexes: WHERE status = 'active' cuts index size by ~40% since most queries filter for active listings
parent_id for nested threads: recursive CTE queries retrieve full comment trees in a single round-trip, with depth capped at 5 levels
Supabase RLS at the database layer: access control enforced in PostgreSQL policies, not application code. No bypass is possible even if the API has a bug
Locale column per listing: single table supports English and Hindi content variations, served based on user preference or Accept-Language header

Technical Deep Dive

Query Performance at 100K+ Rows
PostgreSQL Index Engineering

Unindexed queries on the listings table at 100K+ rows produced 800ms+ response times. The fix was composite index design: a multi-column index on (type, location, created_at) exactly matches the dominant browse query: filter by listing type (job/property/news), filter by location, sort by recency.

A separate partial index WHERE status = 'active' reduces index size by ~40% and speeds up the most frequent user-facing query path, since browsing always filters for active listings. After index optimization, all primary browse queries returned consistently under 200ms even at full table scale. EXPLAIN ANALYZE confirmed sequential scans were eliminated on all hot query paths.

Role-Based Access Control
Supabase RLS Policies

Three user roles: anonymous viewer, authenticated user, and admin. Supabase Row Level Security policies enforce access control directly in PostgreSQL, not in application code. This means access guarantees hold even if the API layer has bugs or is bypassed.

Key policies: authenticated users can only UPDATE listings WHERE user_id = auth.uid(); admins can SELECT from moderation WHERE role = 'admin'; anonymous users can SELECT active listings but cannot INSERT or UPDATE anything. Policies are tested via direct psql sessions impersonating each role, not just through the application flow.

Nested Comment Threads
Recursive CTE + Anonymous IDs

Comments use a nullable parent_id self-referential foreign key for nesting. A recursive CTE (WITH RECURSIVE) retrieves the full comment tree for a listing in a single database round-trip, eliminating N+1 comment fetching. Tree depth is capped at 5 levels via the recursion termination clause to prevent performance degradation on pathological threads.

Anonymous posting uses deterministic per-thread identifiers: a one-way hash of (user_id + listing_id) produces a consistent short tag (e.g., "Anon#7f3a") that lets readers distinguish different anonymous posters in the same thread without any linkable identity across threads.

SEO for Hyperlocal Discoverability
Next.js SSR + ISR

Organic search was the primary user acquisition channel. All listing detail pages use Next.js server-side rendering with full structured metadata: Open Graph tags, JSON-LD schema (JobPosting, RealEstateListing) for rich search results, and canonical URLs with location-specific slugs (/jobs/city/title-slug) that match hyperlocal search intent exactly.

Incremental Static Regeneration (ISR) for listing category pages allows edge-cached responses for the most popular browse views while still revalidating on-demand when new listings appear. The combination of SSR detail pages and ISR browse pages gave the platform both search engine visibility and real-time freshness without choosing between them.

Community Moderation Pipeline
Report Queue + Admin Tooling

At 50K+ users, even a low percentage of bad actors produces a non-trivial absolute volume of harmful content. The moderation system uses a report queue: any authenticated user can flag any listing or comment, creating a moderation record with target_id, target_type, and reason. The admin dashboard surfaces reports sorted by report count and recency for manual review triage.

Admin actions include: hide content (soft-delete from public view), permanent delete, warn user (automated message), and ban. Supabase Realtime pushes new reports to the admin dashboard in real-time, allowing faster response to high-severity content. The entire moderation state is audit-logged for accountability.

Lessons Learned

01
Index design for query patterns, not individual columns
Single-column indexes barely moved the needle. The 200ms target required composite indexes precisely shaped to match the dominant query pattern: type + location + recency. Query plan analysis (EXPLAIN ANALYZE) was essential to verify the right index was being used.
02
Supabase RLS belongs in the database, not the application
Enforcing access control at the PostgreSQL layer rather than the API layer gives guarantees that hold even under bugs, misconfiguration, or API bypass. Test RLS policies by direct psql impersonation of each role, not just through the app.
03
SEO drove more growth than any product feature
The SSR + ISR investment for listing page crawlability compounded significantly over time. Location-specific URL structures made listings discoverable for hyperlocal search queries that national competitors weren't ranking for.
04
Moderation infrastructure before scale, not after
Building the report queue and admin tooling before the platform went wide saved significant community damage. At 50K users, even a 0.1% bad actor rate is 50 accounts - enough to degrade trust quickly without proactive infrastructure in place.
