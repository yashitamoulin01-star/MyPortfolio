Overview

The Business of Professionals (TBoP) is a professional reputation and trust platform built for the Indian market. Professionals create verified profiles, receive peer ratings from colleagues and clients, and accumulate a Trust Score, a composite metric that signals credibility to recruiters and collaborators. Recruiters access a discovery dashboard with filters to find high-trust professionals by domain and experience.

The platform is a multi-tenant SaaS product: professionals subscribe individually, each with strict data isolation enforced at the PostgreSQL layer. Subscription billing runs through Razorpay with webhook-driven subscription state management. The platform sits behind Cloudflare for DDoS protection, WAF rules, and CDN caching.

The Problem

Professional credentialing in India is fragmented and gameable. LinkedIn endorsements can be traded between friends. Certificates can be faked or inflated. There's no system that lets professionals demonstrate trustworthiness through verified, fraud-resistant peer ratings from people who've actually worked with them.

Recruiters face the same problem from the other side: they can't distinguish genuinely reliable professionals from those who've simply optimized their visible credentials. They need a third-party trust signal that's harder to game than self-reported achievements.

Core product challenge: Build a trust system that's useful enough for professionals to pay for, but resistant enough to manipulation that recruiters can rely on it as a genuine signal, not just another vanity metric.

Technical Deep Dive

Multi-Tenant Isolation with PostgreSQL RLS
Data Security

Multi-tenancy on a shared database requires strict data isolation between users. Rather than application-layer filtering (which is fragile and bypassable), TBoP uses PostgreSQL Row Level Security policies that enforce isolation at the database level. Every table has RLS enabled with explicit policies for each user role.

Example policies: professionals can only SELECT/UPDATE their own profile WHERE user_id = auth.uid(); recruiters can SELECT profiles WHERE is_public = true; admins bypass RLS for moderation. Policies were stress-tested via direct psql sessions impersonating each role, not just through the application flow. Edge cases like a user accessing their own rating history vs. another user's were explicitly tested.

Trust Score Algorithm
Fraud-Resistant Reputation Metric

The Trust Score combines four weighted components: peer rating quality (average rating from verified connections, weighted by reviewer credibility score), recency bias (exponential decay function so recent ratings carry more weight than old ones), verification completeness (education, employment, identity verification badges add score multipliers), and platform activity (responsiveness to rating requests, profile completeness).

Fraud resistance comes from coordinated rating pattern detection: if a cluster of accounts rates the same professional within a short time window, with similar reviewer credibility and no historical relationship, the ratings are flagged for manual review before counting toward the score. The algorithm is designed to make gaming it require more effort than simply doing good professional work.

Razorpay Subscription Billing
Payment Infrastructure

Subscription state is the source of truth for feature access. The billing flow: Razorpay Subscription created on plan selection → frontend redirects to Razorpay checkout → on payment, Razorpay fires a webhook event → Supabase Edge Function receives and validates the webhook signature → subscription record upserted in PostgreSQL → user's plan entitlements updated.

Idempotency is critical: webhook events can be delivered more than once. Every webhook handler checks if the event_id has already been processed before making any state changes. This prevents double-billing, double-activation, or double-cancellation from duplicate webhook delivery. The webhook endpoint returns 200 immediately and processes asynchronously to prevent Razorpay timeout retries.

Cloudflare Security Layer
DDoS · WAF · Bot Protection

A reputation platform is a natural target for coordinated attacks: fake account creation, scraping of professional data, and automated rating submission. Cloudflare handles the first line of defense: DDoS mitigation for volumetric attacks, WAF rules for OWASP Top 10 patterns, and Bot Fight Mode for automated rating submission.

Custom WAF rules rate-limit rating submission endpoints per IP and per authenticated user. Cloudflare Analytics flagged suspicious traffic patterns within the first 48 hours of launch. Bot activity that Cloudflare automatically blocked before it reached the origin. The WAF layer proved valuable well before the platform reached significant scale.

Admin Moderation Dashboard
Content Safety + Audit Trail

The moderation dashboard surfaces reported profiles, flagged ratings, and pending verification reviews in a unified queue. Admins see: report count per target, flagged rating patterns, relationship verification status, and the full history of actions taken on each account.

Escalation workflows: Level 1 (automated flag) → Level 2 (admin review) → Level 3 (appeal process). Every admin action writes an audit log entry with admin_id, action_type, target_id, and timestamp, making the moderation history fully auditable and preventing unilateral abusive banning.

Challenges Solved

Multi-tenant data isolation: PostgreSQL RLS policies enforce per-user data boundaries at the database layer, not the application layer, giving guarantees that hold even under API bugs.
Fraud-resistant trust scoring: coordinated rating pattern detection and reviewer credibility weighting make manipulating the Trust Score more effort than earning a genuine reputation.
Idempotent webhook processing: Razorpay webhooks can be delivered multiple times. Every handler checks event_id for prior processing before applying any state changes, preventing double-billing or double-activation.
Zero-downtime deployments: Vercel's atomic deployment model combined with database migration strategy (backward-compatible schema changes first, then code deploy) guarantees no interruption to live subscriptions.
Bot-resistant rating submission: Cloudflare WAF rules rate-limit rating endpoints per IP and per user; Bot Fight Mode catches automated rating submission patterns before they reach the application.
Escalation-grade admin tooling: full audit trail on every moderation action with three-level escalation workflow ensures accountability and prevents both under-moderation and abusive banning.

Lessons Learned

01
Multi-tenant isolation belongs in the database, not the application
Application-layer access control is bypassable under bugs, misconfiguration, or direct database access. RLS policies enforced in PostgreSQL hold unconditionally. Test them by impersonating each role in psql, not just through the app.
02
Webhook idempotency is non-negotiable in payment systems
Razorpay webhooks can arrive multiple times for the same event. Processing a payment event twice can double a user's subscription, credit, or entitlement. Always check event_id for prior processing before applying any state change - and respond 200 immediately to prevent retry loops.
03
Design trust algorithms adversarially from day one
Before writing the Trust Score algorithm, I spent a day trying to game it. Every exploit I found got a countermeasure built in. Rating systems that aren't designed with adversarial scenarios in mind get gamed within weeks of launch.
04
Cloudflare WAF catches real attacks faster than you expect
Bot activity was detectable in Cloudflare Analytics within 48 hours of launch - well before any manual detection would have caught it. For any platform handling user reputation or financial data, a WAF layer is infrastructure, not a luxury.
